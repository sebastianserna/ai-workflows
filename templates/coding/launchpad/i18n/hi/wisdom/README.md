# wisdom/ — ज्ञान आधार

इस फ़ोल्डर में प्रोजेक्ट का तकनीकी दस्तावेज़ीकरण है। यह **प्रामाणिक स्रोत** है जिससे AI एजेंट निर्णय लेने से पहले परामर्श करते हैं।

## उद्देश्य

- आर्किटेक्चर निर्णयों और उनके तर्कों का दस्तावेज़ीकरण करना
- तकनीकी गाइड को अद्यतन रखना
- एजेंट को गलतियाँ दोहराने या असंगत निर्णय लेने से रोकना
- ऐसे ज्ञान को संरक्षित करना जो कोड में समा नहीं सकता

## संरचना

```
wisdom/
├── README.md           # यह फ़ाइल
├── architecture/       # सिस्टम कैसे डिज़ाइन किया गया है
├── development/        # प्रोजेक्ट पर कैसे काम करें
└── operations/         # कैसे डिप्लॉय और ऑपरेट किया जाता है
```

### architecture/

आर्किटेक्चर निर्णय, सिस्टम संरचना, चुने गए पैटर्न और उनके तर्क। **"यह कैसे डिज़ाइन किया गया है और क्यों"** का उत्तर देता है।

उदाहरण: फ़ोल्डर संरचना, लेयर पैटर्न, लाइब्रेरी निर्णय, कंपोनेंट डायग्राम।

### development/

दैनिक डेवलपमेंट गाइड। **"इस प्रोजेक्ट पर कैसे काम करें"** का उत्तर देता है।

उदाहरण: कोड कन्वेंशन, टेस्टिंग गाइड, git वर्कफ़्लो, लोकल सेटअप, लिंटिंग।

### operations/

कोड को प्रोडक्शन में ले जाने और चालू रखने से संबंधित सब कुछ। **"कैसे डिप्लॉय और ऑपरेट किया जाता है"** का उत्तर देता है।

उदाहरण: CI/CD पाइपलाइन, एनवायरनमेंट कॉन्फ़िगरेशन, मॉनिटरिंग, एनवायरनमेंट वेरिएबल।

## और फ़ोल्डर कब जोड़ें

तीन आधार फ़ोल्डर अधिकांश प्रोजेक्ट को कवर करते हैं। यदि प्रोजेक्ट किसी विशिष्ट डोमेन में बढ़ता है, तो समर्पित फ़ोल्डर जोड़ें:

| फ़ोल्डर | कब जोड़ें |
|---------|----------|
| `database/` | प्रोजेक्ट में माइग्रेशन के साथ अपना डेटाबेस है |
| `security/` | Auth नियम, अनुमतियाँ, या कंप्लायंस दस्तावेज़ीकरण आवश्यक है |
| `integrations/` | एकाधिक बाहरी API या तृतीय-पक्ष सेवाएँ हैं |
| `api/` | सार्वजनिक API जिसे अन्य उपयोग करते हैं |

**नियम:** "बस ऐसे ही" खाली फ़ोल्डर न बनाएँ। उन्हें तब बनाएँ जब अंदर रखने के लिए कम से कम एक वास्तविक दस्तावेज़ हो।

## कन्वेंशन

- Markdown में दस्तावेज़, प्रति फ़ाइल एक विषय
