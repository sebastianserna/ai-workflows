# wisdom/ — ナレッジベース

このフォルダにはプロジェクトの技術ドキュメントが含まれています。AIエージェントが意思決定前に参照する**信頼できる情報源**です。

## 目的

- アーキテクチャの決定事項とその理由を記録する
- 技術ガイドを最新の状態に保つ
- エージェントが同じ間違いを繰り返したり、一貫性のない決定を下すことを防ぐ
- コード自体では表現できない知識を保存する

## 構成

```
wisdom/
├── README.md           # このファイル
├── architecture/       # システムの設計方法
├── development/        # プロジェクトでの作業方法
└── operations/         # デプロイと運用方法
```

### architecture/

アーキテクチャの決定事項、システム構造、選択したパターンとその理由。**「どのように設計されており、なぜそうなのか」**に答えます。

例：フォルダ構成、レイヤーパターン、ライブラリの選択、コンポーネント図。

### development/

日々の開発ガイド。**「このプロジェクトでどのように作業するか」**に答えます。

例：コード規約、テストガイド、Gitワークフロー、ローカルセットアップ、リンティング。

### operations/

コードを本番環境にデプロイし、稼働させ続けるためのすべて。**「どのようにデプロイし運用するか」**に答えます。

例：CI/CDパイプライン、環境設定、監視、環境変数。

## フォルダを追加するタイミング

3つの基本フォルダでほとんどのプロジェクトをカバーできます。特定のドメインでプロジェクトが拡大した場合、専用フォルダを追加します：

| フォルダ | 追加するタイミング |
|----------|-------------------|
| `database/` | プロジェクトにマイグレーション付きの独自DBがある場合 |
| `security/` | 認証ルール、権限、コンプライアンスの文書化が必要な場合 |
| `integrations/` | 複数の外部APIやサードパーティサービスがある場合 |
| `api/` | 他者が利用する公開APIがある場合 |

**ルール：** 「念のため」空フォルダを作成しない。中に入れる実際のドキュメントが少なくとも1つある時に作成する。

## 規約

- Markdownでドキュメントを作成、1ファイルにつき1トピック
- アーキテクチャの変更時に更新を維持
- `AGENTS.md` の「詳細仕様」セクションから参照

## workplan/ との違い

| | `wisdom/` | `workplan/` |
|---|-----------|-------------|
| **含まれるもの** | 永続的なドキュメント | ライフサイクルのある作業計画 |
| **変更タイミング** | アーキテクチャの進化時 | 常時（backlog → coding → done） |
| **フォーマット** | 自由形式 | ヘッダー付きの標準テンプレート |
| **目的** | システムを理解する | やるべきことを管理する |
