# wisdom/ — 知识库

此文件夹包含项目的技术文档，是 AI 代理在做出决策前参考的**权威来源**。

## 目的

- 记录架构决策及其原因
- 保持技术指南的更新
- 防止代理重复犯错或做出不一致的决策
- 保存代码本身无法承载的知识

## 结构

```
wisdom/
├── README.md           # 本文件
├── architecture/       # 系统设计方式
│   └── AGENTS.md
├── development/        # 项目开发方式
│   └── AGENTS.md
└── operations/         # 部署和运维方式
    └── AGENTS.md
```

每个 `AGENTS.md` 为 AI 代理提供关于文件夹范围及何时查阅的上下文信息。

### architecture/

架构决策、系统结构、所选模式及其原因。回答**"系统是如何设计的，为什么这样设计"**。

示例：文件夹结构、分层模式、库选择、组件图。

### development/

日常开发指南。回答**"我如何在这个项目上工作"**。

示例：代码约定、测试指南、Git 工作流、本地环境搭建、代码检查。

### operations/

与将代码部署到生产环境并保持运行相关的一切。回答**"系统是如何部署和运维的"**。

示例：CI/CD 流水线、环境配置、监控、环境变量。

## 何时添加更多文件夹

三个基础文件夹适用于大多数项目。如果项目在特定领域有所增长，可添加专用文件夹：

| 文件夹 | 何时添加 |
|--------|----------|
| `database/` | 项目拥有自己的数据库和迁移 |
| `security/` | 需要记录认证规则、权限或合规要求 |
| `integrations/` | 多个外部 API 或第三方服务 |
| `api/` | 供他人使用的公共 API |

**规则：** 不要"以防万一"而创建空文件夹。只在有实际文档需要放入时才创建。

## 约定

- 使用 Markdown 编写文档，每个文件一个主题
- 架构变更时及时更新
- 在 `AGENTS.md` 的"详细规范"部分引用

## 与 workplan/ 的区别

| | `wisdom/` | `workplan/` |
|---|-----------|-------------|
| **包含** | 永久性文档 | 具有生命周期的工作计划 |
| **变更** | 架构演进时 | 持续变更（backlog → coding → done） |
| **格式** | 自由格式 | 带标题的标准化模板 |
| **目的** | 理解系统 | 管理待办事项 |
